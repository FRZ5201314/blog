# xlua 热修复浅尝

项目中使用到了 loxodon-framework 框架应用 MVVM，用以实现数据绑定等相关功能。

由于项目发布与 webgl 平台，包体过大会导致加载场景需要花费很长的时间。
在贴图优化模型优化等一系列优化后包体大小依旧超过百兆
于是引入 addressable 来做资源热更。
运行时动态加载和释放资源，既能减轻一波运行内存占用过多导致帧数较低的痛，又能减少首次进入场景所需要花费的漫长等待时间。

然而在使用资源热更之后呢，决定再研究一下代码热更。

于是就开始了解代码热更相关的方案

# 代码热更方案分析：

InjectFix：c# 热修复，优点多，体积小。但是 demo 测试未通过！github 的 issues 有 99+，且作者没回复！ 【gg】

ILRuntime: c# 热更新，基于热更模块的更新，分工程分程序集（有手段可以不分工程），但是对于老的现有项目它无法直接访问原类的私有成员，需要做较多的修改（适配器）。【不适合做热修复，新项目用它来从头开发或许可以，同样对于它的特性、重定向、CLR 绑定、委托转换器、跨域继承等需要学习】

Xlua: lua 热修复，现有项目在不需要动原来代码的情况下，只需要注入依赖并定义指定热修复命名空间或类型就可以通过 lua 脚本更新完成热修复。【需要对 lua 语法和特性的学习】

CSHotfix，XIL：c# 热修复，基于 ILRuntime，还是那个问题，它们访问原类的私有变量之类是要特殊的写法，一个类成员方法往往避免不了这些写法，因此你要修复一个大的函数，对原来逻辑的重新实现往往是无可避免的。

# 热更方案分析总结：

xlua 全网使用较多相关技术文章多于 ILRuntime，学习时间成本对比 ILRuntime 差不了多少。xlua 新项目和老项目用起来没有区别，新项目也可以逻辑就用 c# 来写，然后对于小的修改可以通过 xlua 打补丁，避免重新发布 webgl 包，大的修改变动就用 c# 写再发包。 ILRuntime 用热修复方案，因为热更程序集的代码和原生程序集的写法是有不同的，所以大的更新，还是原生 untiy 工程里更新更合适，为了减少发布次数而编写与原生不同写法的热更程序集需要花费更多的时间，显然得不偿失。ILRuntime 做新项目，和 xlua 一样都是需要一定的学习成本的。另外，现在 ILRuntime 作者自己也是倾向于 xlua，在 demo 中加入了 CsharpToXLua 示例。

整体而言。对于 webgl 项目，我需要的只是适用于处理小问题的代码热修复方案。大的修改变动还是使用 c# 正常开发并重新发布 webgl。

最终决定使用 xlua 代码热修复方案。

# 项目中实际应用：

  1. 使用中遇到的问题及解决办法

    xlua 的 [Plugins] 文件夹必须是在项目 Assets 文件夹的绝对路径下，不可以放在 Assets 的其他文件夹下，否则会导致编译报错发布失败。

    热修复需要调用基类的方法时不再使用 xlua.hotfix 而是改用 util.hotfix_ex 进行热修复，调用基类方法使用 self:baseFuncTest ()

    xlua 使用 async await 时，发现类似 GetAwaiter 等多数泛型方法在 xlua 无法调用，必须对泛型方法进一步封装为指定泛型类型的方法，才能正常调用。

    目前项目已发布代码热更版本并使用热更稳定正常。后续遇到问题再更新记录…
